# CTP Credit-Modelling Feature Pipeline (Synthetic Assessment)

Point-in-time-correct feature engineering pipeline preparing CTP telco data
for CF credit scoring, per `CANDIDATE_BRIEF.md`. All data is synthetic.

## Setup

```bash
python -m venv .venv && source .venv/bin/activate   # optional but recommended
make setup
```

This installs `pandas`, `numpy`, `pyyaml`, `pyarrow`, and `pytest` (see
`requirements.txt`). Requires Python 3.10+.

Place the provided raw CSVs in `data/raw/` and the reference files in
`data/reference/` if not already present (see `data/raw/` and
`data/reference/` for the expected filenames — they match the brief's
dataset list exactly).

## Run

```bash
make run
```

equivalent to:

```bash
python -m src.pipeline
```

This loads all 7 raw tables, runs Task 1 data-quality checks, and builds
the Task 2 point-in-time feature table. Outputs are written to `outputs/`:

- `dq_report.json` — machine-readable DQ findings
- `dq_summary.md` — human-readable DQ narrative
- `feature_table.csv` / `feature_table.parquet` — one row per `application_id`

## Test

```bash
make test
```

equivalent to:

```bash
pytest -v
```

`tests/test_pipeline.py` asserts: no leakage columns in the output, exactly
one row per `application_id`, point-in-time correctness of the trickiest
features (billing payment state, usage same-day exclusion, snapshot as-of
join), and that the DQ checks actually catch the defects present in this
dataset.

See `docs/technical_note.md` for design decisions, point-in-time handling
(including two non-obvious leakage traps found in `billing_monthly` and
`crm_events`), production scaling notes, monitoring plan, and known
limitations. See `docs/feature_dictionary.md` for every output column's
definition, source, transformation, null handling, and governance
classification. See `docs/cf_handoff_contract.md` for the CTP→CF governed
handoff terms.
